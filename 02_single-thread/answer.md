# 答え：シングルスレッドのボトルネック（CPU追加しても意味がない）

## どこを見るか

### ステップ1：topの全体CPU vs プロセスの乖離に気づく

```
%Cpu(s): 24.8 us,  0.2 sy,  0.0 ni, 74.9 id ...
                                     ^^^^^^^^ 全体の75%はアイドル

    PID USER  ...  %CPU  COMMAND
   7823 batch ...  99.9  python3 aggregate.py
                   ^^^^  1プロセスだけ100%
```

全体CPU使用率25%なのに、1プロセスだけ99.9%。  
4コアなら「1コア100% ÷ 4コア = 全体25%」で計算が合う。

### ステップ2：mpstatでコアごとを確認

```
Average:     CPU    %usr   %idle
Average:     all   25.00   74.83  ← 全体は余裕あり
Average:       0   99.34    0.00  ← CPU0だけ100%張り付き
Average:       1    0.33   99.67  ← ほぼ寝てる
Average:       2    0.33   99.67  ← ほぼ寝てる
Average:       3    0.33   99.67  ← ほぼ寝てる
```

CPU0だけ100%。他のコアはほぼ遊んでいる。  
= シングルスレッドで動いているプロセスがCPU1個を使い切っている。

## なぜ間違えるか

topを1画面だけ見て「CPU 25%で余裕あり」と判断してしまう。  
**「サーバーのスペックを上げればいい」という結論は間違い。**  
コアを4→8に増やしても、シングルスレッドのプログラムはCPU1個しか使えない。

## 解決の方向性

- `aggregate.py` をマルチスレッド/マルチプロセスに改修する
- 集計対象データを分割して並列実行する
- Pythonの場合はGILの影響でマルチスレッドが効かないことがあるため、multiprocessingモジュールの使用を検討
- CPUバウンドな処理ならGo/Rustへの書き直しも選択肢
