# 答え：スワップスラッシング（メモリが足りなくてswapとの間でデータを往復させている）

## どこを見るか

### ステップ1：free のトラップを見抜く

```
              total        used        free      shared  buff/cache   available
Mem:           7.6G        6.2G        523M        234M        923M        876M
Swap:          4.0G        3.7G        300M
                           ^^^
```

`free`（空き）は523MBあるように見える。しかし注目すべきは`Swap: 3.7G used`。  
4GBのswapのほぼ全部を使っている。

`available`は876MB。これが実際に使える量の目安だが、swapが満杯に近い状態で  
パフォーマンスは既に限界を超えている。

### ステップ2：vmstatの si/so を見る

```
 r  b   swpd   free   buff  cache   si   so    bi    bo
 2  1 3878912 535804  ...         487  512   523   548
 1  2 3879034 534912  ...         523  498   567   523
                                  ^^^  ^^^
                              swap-in  swap-out
```

- `si`（swap in）：ディスクからメモリにswapを読み込んでいる量（KB/s）
- `so`（swap out）：メモリからディスクにswapを書き出している量（KB/s）

**si=487〜523、so=498〜534 が毎秒続いている** = スラッシング状態。  
メモリとswap間でデータを往復させ続けていて、CPUがほとんどI/O待ちに使われている。

### ステップ3：swpd の増加を確認

```
vmstat の swpd 列:
3878912 → 3879034 → 3879128 → 3879234 → 3879312
```

毎秒swp使用量が増え続けている。まだswapを食い続けている。

## なぜ間違えるか

`free` を見て「Mem: 523M空いてる、まだ余裕がある」と判断する。  
しかし実際は3.7GBのswapを使い切りそうになっており、  
**swapとメモリ間のI/Oがシステム全体を遅くしている。**

## 解決の方向性

**即時対応**
- 夜間バッチ（`nightly.py`）の同時実行を止めて、javaのメモリを空ける
- `swapoff -a && swapon -a` でswapをフラッシュ（要空きメモリ確認）

**根本対応**
- メモリ増設（8GB→16GB以上）
- Javaのヒープサイズ上限（-Xmx）を見直す
- バッチと常駐プロセスが同時にピークメモリを使わないようにスケジュールを分ける
