# 答え：inodeが枯渇している（容量ではなくファイル数の上限）

## どこを見るか

### ステップ1：df -h では何も見えない（これがトラップ）

```
/dev/sdc1       100G   42G   53G  45% /var
```

容量は53GB余っている。普通ならここで「ディスクは問題なし」と判断してしまう。

### ステップ2：df -i を見る

```
Filesystem       Inodes   IUsed    IFree IUse% Mounted on
/dev/sdc1       6553600 6553599        1  100% /var
                                       ^  ^^^
                                       残り1個  100%
```

`/var` のIUse%が100%。inodeが1個しか残っていない。

## inodeとは何か

Linuxのファイルシステムは、ファイルの「中身（データ）」と「管理情報（inode）」を別々に持っている。  
inodeにはファイルサイズ・所有者・タイムスタンプ・データの場所などが記録されている。

- ディスク容量が余っていても → データを書ける場所がある
- inodeが枯渇すると → 新しいファイルの「管理情報」を置けない → ファイルが作れない

## なぜ起きるか

ログコレクターが大量の小さいファイルを `/var/log/app/` に作り続けていた。  
1ファイルにつき1 inodeを消費するため、数百万個のファイルが溜まるとinodeが枯渇する。

## 解決の方向性

**即時対応**
```bash
# どのディレクトリにファイルが集中しているか確認
find /var -xdev -printf '%h\n' | sort | uniq -c | sort -rn | head -20
```

**根本対応**
- ログローテーション設定の見直し（古いファイルを定期削除）
- ログを1ファイルにまとめる設計に変更（細かく分割しすぎない）
- ファイルシステム作成時に `-i` オプションでinode数を増やす（再作成が必要）
