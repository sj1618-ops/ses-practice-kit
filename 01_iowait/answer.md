# 答え：原因はディスクI/O（mysqldがI/O待ちで詰まってる）

## どこを見るか

### ステップ1：topの「wa」に気づく

```
%Cpu(s):  2.3 us,  0.5 sy,  0.0 ni,  1.8 id, 95.2 wa,  0.0 hi,  0.2 si,  0.0 st
                                               ^^^^^^^^
```

`wa`（iowait）が95.2%。CPUの稼働率ではなく、CPUがI/O完了を待っている時間の割合。  
`us`（user）は2.3%しかない。CPUが忙しいわけではない。

### ステップ2：topのプロセス状態「D」を確認

```
   4821 mysql     20   0  985284 412345  18234 D  99.7   5.1  18:42.33 mysqld
                                               ^
```

`S`（sleep）ではなく`D`（uninterruptible sleep）。  
I/O完了を待ってブロックされている状態。

### ステップ3：vmstatで確認

```
 r  b   swpd   free   buff  cache   si   so    bi    bo   in   cs us sy id wa st
 3  5      0 326148  46782 ...       0    0 28432   128 ...            2  1  2 95  0
    ^                                          ^^^^^
```

- `b`（blocked）が5〜6：I/O待ちでブロックされているプロセス数
- `bi`（blocks in）が28000超：大量のディスク読み込みが発生中

### ステップ4：iostatで犯人ディスクを特定

```
Device   r/s     rkB/s   r_await  %util
sdb    423.45  54287.23   47.82   98.72  ← これ
```

- `sdb`の`%util`が98.72%：ほぼ100%使い切っている
- `r_await`が47.82ms：1回のread要求に47msかかっている（正常は1〜5ms）

## なぜ間違えるか

topの1行目に「load average: 3.87」と出て、プロセスのCPU欄にmysqldが99.7%と出ているため、  
**「CPUが問題だ」と思い込む。**

しかし実際はmysqldがディスクI/Oを待っているだけで、CPUは全然使っていない。

## 解決の方向性

- mysqlのスロークエリログを確認してフルスキャンのクエリを特定
- sdbのI/Oがなぜ増えているか調査（定期バッチ？インデックス欠落？）
- ディスクがHDDなら、SSDへの移行を検討
